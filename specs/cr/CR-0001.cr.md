## Summary

Implementar el pipeline completo del spike_gpu de punta a punta: dado un URL de YouTube, descargar audio (yt-dlp), transcribir (faster-whisper medium), obtener metadata y comentarios (YouTube API v3), analizar con Ollama qwen2.5:7b (puntos clave, temas, tono, sentimiento de comentarios), generar guion estructurado con prompts de imagen y video secuenciales, medir tiempos por fase, persistir resultados en PostgreSQL, y exponer todo via UI Streamlit y endpoint FastAPI.

## Intent

El spike demuestra viabilidad técnica de un pipeline de análisis de video en GPU cloud (RunPod) versus local, midiendo tiempos por fase en ambos entornos para informar la decisión de escalar al proyecto Transcriptor_Angeik. Éxito = pipeline corre de punta a punta con un URL de YouTube real, produce guion + prompts, y registra tiempos en `tiempos_ejecucion` con columna `environment` que permite el versus local/cloud.

## Assessment

**Blast radius:** Cero — proyecto greenfield, sin código existente. Todos los archivos son nuevos.

**Reversibility:** Fácil — no hay migraciones sobre datos existentes, no hay sistema en producción. Revertir = eliminar archivos.

**Security:** Sin impacto de seguridad crítico. API keys (YouTube API key, DB credentials) se almacenan en `backend/.env` (gitignored). El alcance es local/dev — no hay auth de usuarios, no hay multi-tenancy, no hay datos sensibles de terceros.

**Dependencies:** Ninguna CR previa. Servicios externos requeridos: Ollama corriendo localmente en localhost:11434, PostgreSQL via Docker Compose, YouTube Data API v3 key configurada.

**Unknowns al momento del intake:** Ninguno relevante — arquitectura, stack y decisiones de diseño están completamente definidos en ARCHITECTURE.md y CLAUDE.md.

## Initial Risks

| Riesgo | Probabilidad | Mitigación |
|--------|-------------|------------|
| Ollama qwen2.5:7b no disponible localmente | Media | `OLLAMA_BASE_URL` configurable via `.env`; tests mockean LLM |
| YouTube API quota excedida en desarrollo | Baja | Cache en BD: una vez procesado, no re-llama a la API |
| faster-whisper con CUDA no disponible en máquina dev | Media | `WHISPER_DEVICE=cpu` en `.env` local; VAD filter funciona en ambos |
| Transcripciones largas exceden contexto del LLM | Alta (conocido) | Chunking por 3000 palabras + síntesis 2 pasadas — diseñado para esto |
| Docker Compose no corriendo al ejecutar pipeline | Baja | Health check en `GET /health`; error explícito si BD no conecta |

## Recommended Approach

Implementar capa por capa de adentro hacia afuera siguiendo el orden de dependencias:

1. **Database** — schema SQL (5 tablas) + psycopg3 pool
2. **Config** — `settings.py` con python-dotenv
3. **Core** — `timer.py` → `downloader.py` → `transcriber.py` (sin dependencias entre ellos)
4. **Services** — `schemas.py` → `youtube_service.py` → `analyzer.py` → `generator.py`
5. **Pipeline** — `pipeline.py` orquestador (ensambla todo lo anterior)
6. **API** — `server.py` FastAPI con POST /analyze, GET /health
7. **UI** — `app.py` Streamlit

Este orden permite testear cada capa antes de construir la siguiente. Los servicios LLM se mockean en tests para no depender de Ollama en CI.
